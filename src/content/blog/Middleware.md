# 什么是分布式
## 概念介绍
分布式系统：一组通过网络进行通信，为了完成共同的任务而协调工作的计算机节点组成的系统。比如：两个程序分别在两台计算机上，相互协作完成同一个服务（或者功能），从理论上讲，这两个程序组成的系统，就可以称作“分布式系统”。

集中式系统：所有的程序、功能都集中到一台主机上，对外提供服务。

那么为什么产生了分布式系统？
	单个机器无法完成业务，比如巨量数据、同时的多次访问等。
## 分布式系统特性
**内聚性**：每个节点高度自治
**透明性**：系统对用户来说是透明的，用户无法感知系统如何实现
**可拓展性**：
1. 垂直扩展：优化系统的性能或者升级硬件。
	1. 易实现
	2. 成本高
	3. 有单点失效的可能
2. 水平扩展：增加计算单元扩展系统的规模
	1. 成本低
	2. 更可靠
	3. 较难实现
**可用性**：在外部资源得到保证的前提下，系统在规定的条件下或规定的时间内处于可执行状态的能力。
通俗的讲，就是在没有意外状况的前提下（搞活动，突然高并发），能够保证正常状态
**可靠性**：在给定时间和给定条件下，系统能正确执行功能的概率
**高性能**：高并发、低延迟
**一致性**：为了提高可用性和可靠性，一般会引入冗余。为了保证节点的状态一致，需要解决一致性问题。
## 分布式系统常见问题
### 网络
1. 消息丢失
2. 网络分区：两个机房内部通信正常，机房之间通信中断
3. 消息乱序
4. 数据错误：通常使用校验码进行简单的检查
5. 分布式系统三态：成功、失败、超时（未知）
### 节点故障
系统需要监控节点的状态，节点故障的情况下把该节点负责的任务转移到其他节点

## CAP理论
一致性(Consistency)：即原子性，对写操作返回成功，所有的读操作都要读取到新数据；返回失败，所有的读操作都不能读到。
可用性(Availability)：用户的每个请求都能接受到一个响应，无论成功还是失败。要求系统不能有单点的存在。
分区容忍性(Partition Tolerance)：即除了整个网络出现故障外，其它的故障都不能导致整个系统无法正确响应。
	分布式系统中遇到网络故障，可能会形成分区。如果存在单点问题，即一个数据只在一个节点中出现，这时就不能分区容忍，要提高容忍性就要进行冗余（备份）。但是进行冗余，又会导致一致性的问题，因为同时需要复制、更新的数据变多了，可用性就会降低。
**CAP - 鱼与熊掌不可兼得**
CA - without P：分布式系统中，分区无法完全避免。分区出现的时候，还需要在C和A中做出选择：选择一致性则需要等待分区回复，在此期间牺牲可用性；选择可用性，则无法保证各个分区数据的一致性。
CP - without A：放弃 A（可用性），一旦分区发生，部分节点之间失去联系，为了保证一致性，需要等待受影响的服务所在的节点数据一致（本质上就是等待分区恢复），而等待期间无法对外提供服务。如此，CP 也是可以保证的，很多传统的数据库分布式事务都属于这种模式。
AP - without C：放弃 C（一致性），保证高可用并允许分区。一旦分区发生，节点之间可能会失去联系，为了高可用，每个节点只能用本地数据提供服务，而这样会导致全局数据的不一致性。
## 分布式中间件 - 缓存
缓存能够提高系统响应速度、改善系统运行性能的技术。缓存通常基于内存，传统的DB从磁盘读数据。
1. 高性能：分布式缓存将高速内存作为数据对象的存储介质，数据以key/value形式存储
2. 动态扩展性：支持弹性扩展，通过动态增加或减少节点应对变化的数据访问负载，提供可预测的性能与扩展性，同时最大限度的提高资源利用率
3. 高可用性：包含数据可用性与服务可用性。基于冗余机制实现高可用性，无单点失效问题，支持故障的自动发现，透明地实施故障切换，不会因服务器故障而导致缓存服务中断或数据丢失，动态扩展时自动均衡数据分区，同时保障缓存服务持续可用。
4. 易用性：提供单一的数据与管理视图；API 接口简单且与拓扑结构无关；动态扩展或失效恢复时无需人工配置；自动选取备份节点；多数缓存系统提供了图形化的管理控制台，便于统一维护。

典型应用场景：
1. 页面缓存：缓存Web页面的内容片段，html、css、图片等
2. 应用对象缓存：缓存系统作为ORM框架的二级缓存对外提供服务，减轻数据库的负载压力，加速应用访问
3. 状态缓存：包括session会话状态及应用横向扩展的状态数据，这类数据难以恢复，对可用性要求较高，多用于高可用集群
4. 并行处理：涉及大量中间计算结果需要共享
5. 事件处理：分布式缓存提供了针对事件流的连续查询处理技术，满足实时性需求
## 分布式中间件 - 锁
多线程并发场景，需要保证共享资源在同一时刻只能被同一个线程操作。引入锁机制，如Java中的synchronized和Lock进行实现。然而分布式多进程、多线程分布在不同节点，需要一种跨JVM的互斥机制进行控制访问。
### 预备知识
**原子性问题**：一个操作不可中断，每次修改必须同步到所有现场。由于执行不连贯导致的数据不一致问题被称作原子性问题。
**可见性问题**：可见性问题的出现与线程访问共享数据的方式有关，一个线程对共享数据的修改不能立即为其它线程所见。
**有序性问题**：为了提高执行效率，CPU 会对那些没有依赖关系的指令重新排序，重新排序后的执行结果与顺序执行结果相同。
如：
```java
int a = 0;
int b = 1;
```
单线程中，指令重排在单线程环境下安全，但是多线程环境下可能有问题。
```java
线程A：
String str = "Hello World"; //指令1
boolen flag = false; //指令2

线程B：
while(flag)
{
	doSome();
}
str.toUpperCase(); //指令3
```
如果A顺序执行，B的执行不会出现问题。指令重排后，假如线程A先执行指令2，切换到线程B，终止循环，执行指令3，此时线程A的str还没有创建，出现空指针异常。
### 锁机制
Java中：如 volatile 关键字，可以禁止指令重排，从而避免有序性的问题；
对于原子性问题和可见性问题，在高并发场景下，为了保证共享资源在同一时刻只能被同一个线程执行，需要对公共资源进行互斥控制，即加锁。
Java 提供了两种内置的锁的实现：由 JVM 实现的 synchronized 和 JDK 提供的 Lock

synchronized 本质上是通过锁来实现的。对于同一个代码块，为了实现多个线程在一个时刻只能有一个线程执行，需要在某个地方做个标记，每个线程都必须能看到它。取消标记后，别的线程再设置标记。
Lock 接口的实现类的基本思想是，用一个 volitile 修饰的 int 型变量，保证每个线程都能拥有对该 int 的可见性和原子修改。
实现分布式锁需要依赖一个第三方系统，常见的分布式锁实现方式有三种：数据库、缓存、分布式协调系统。
## 分布式中间件之消息队列
特点：
1. 速度快、高吞吐量、分布式、多分区
2. 支持动态扩展
3. 将数据持久化到硬盘以及副本防止数据丢失
4. 支持多消费者
5. 支持Online(实时消费)和Offline(离线消费、比如按天消费)的场景
### 典型应用场景
**异步处理**
如：用户注册后需要发注册邮件和注册短信。
传统的做法有：
1. 串行方法：注册信息写入数据库 -> 发送注册邮件 -> 发送注册短信
2. 并行方法：注册信息写入数据库后 同时发邮件和短信
异步处理的话，注册信息写入数据库后，写入消息队列后，直接给用户返回成功。然后邮件系统和短信系统再进行操作。

**应用解耦**
如：电商架构，客户在电商网站下订单 -> 调用库存系统接口
存在风险：
1. 假如库存系统无法访问，订单减库存将失败，导致订单失败
2. 短时间大量请求，频繁查询库存，修改库存，库存系统负载很大
引入消息队列后，订单系统写入消息队列后，返回下单成功。消息队列提供异步的通信协议，保存到被接受者取出。
库存系统中，从消息队列中获取下单信息，然后根据下单信息进行库存操作

**流量削峰**
商家搞活动时，通常会出现流量暴增，外部请求超过系统处理能力
这时引入消息队列，缓解短时间内高流量压力
1. 用户的秒杀请求，服务器接收后，首先写入消息队列，然后返回成功。假如消息队列长度超过最大数量，直接抛弃用户请求或跳转到失败页面
2. 秒杀业务根据消息队列中的请求信息，再做后续处理

**消息通讯**
1. A通过客户端发送消息到群里，服务器端将消息写入消息队列
2. 消息队列，负责消息数据的接收，存储和转发
3. B通过客户端查看群消息，订阅并消费消息队列中的信息

# 主流分布式缓存方案的解读及比较
分布式缓存一般被定义为一个数据集合，把数据分布到任意数目的集群节点上。集群中的一个具体节点负责缓存中的一部分数据，对外提供统一的访问接口。
## Redis
### Redis能做什么
开源的，基于内存存储 也可以持久化的Key-Value，可用作数据库、告诉缓存、锁和消息队列。
作为缓存的常见业务场景有：
1. 缓存热点数据，减轻数据库负载；
2. 基于 List 结构显示最新的项目列表；
3. 基于 Sorted Set 来做排行榜，取 Top N；
4. 基于 Set 来做 uniq 操作，如页面访问者排重；
5. 基于 Hset 做单 Key 下多属性的项目，例如商品的基本信息、库存、价格等设置成多属性。
### Redis特点
1. Redis支持数据的持久化，可以将内存中的数据保存在磁盘中，性能和可靠性兼顾
2. Redis不仅支持简单的key-value，还支持字符串、列表、集合等数据结构
3. Redis支持数据的备份，即 Master-Slave 模式，Master 故障时，对应的 Slave 将通过选举升主，保障可用性
4. 主进程是单线程工作，因此Redis所有操作都是原子性的。单个操作时原子性，多个操作也支持事务
5. Redis性能优越，读写速度快，key-value的大小限制也很高
除了上述特点，Redis 还支持 Publish/Subscribe、通知、Key 老化逐出等特性。
#### Redis数据淘汰策略
Redis 支持配置最大内存，当内存不够用时，会通过淘汰策略来回收内存，Redis 提供了丰富的淘汰策略，粒度粗细皆有，适用多种应用场景。
- `volatile-lru`：从已设置过期时间的数据集中挑选最近最少使用的数据淘汰；
- `volatile-ttl`：从已设置过期时间的数据集中挑选将要过期的数据淘汰；
- `volatile-random`：从已设置过期时间的数据集中任意选择数据淘汰；
- `allkeys-lru`：从数据集中挑选最近最少使用的数据淘汰；
- `allkeys-random`：从数据集中任意选择数据淘汰；
- `no-enviction`：当内存使用达到阈值的时候，所有引起申请内存的命令会报错。
#### Redis内存管理
Redis 使用 C 语言编写，但为了提高内存的管理效率，并没有直接使用 malloc/free 函数，Redis 默认选择 jemalloc 作为内存分配器，以减小内存碎片率。
Redis 为 Key-Value 存储定制了两种对象，其中 Key 采用 SDS（Simple Dynamic String)，Value 采用 redisObject
### Redis高可用解决方案
#### 方案一：Redis Cluster
Redis Cluster可线性扩展到1000个节点。采用的是无中心架构，每个节点都保存数据和整个集群状态，都和其他所有节点连接，客户端直连Redis服务，免去了Proxy代理的损耗。
Redis Cluster最小集群需要三个主节点，为了保障可用性，每个主节点最少挂一个从节点
![redis-cluster](Redis-Cluster.png)
#### 方案二：Twemproxy
一个使用 C 语言编写、以代理的方式实现的、轻量级的 Redis 代理服务器。对后端的多台Redis实例进行统一管理，不用关心具体多少个真实的Redis实例。
当某个节点宕掉时，Twemproxy 可以自动将它从集群中剔除，而当它恢复服务时，Twemproxy 也会自动连接。由于是代理，Twemproxy 会有微小的性能损失。
![Twemproxy](Twemproxy.png)
#### 方案三：Codis
Codis 是一个分布式 Redis 解决方案，对于上层的应用来说，连接到 Codis Proxy 和连接原生的 Redis Server 没有明显的区别（部分命令不支持）， 上层应用可以像使用单机的 Redis 一样使用，Codis 底层会处理请求的转发，不停机的数据迁移等工作。
## Memcached
### 简介
Mem+cache 内存+缓存
一种基于内存的Key-Value存储系统，单个Value最大1MB，适用于存储小块的任意数据
适用的业务场景有：
1. 缓存动态网页，以及一些很小但被频繁访问的数据、文件；
2. 缓存 Session 数据 、临时数据；
3. 缓存计算结果和渲染后的网页模板。
### 特点
1. 基于内存存储，速度快，对内存的要求高，CPU要求低，不支持持久化
2. 容量达到指定值后，将基于LRU(Least Recently Used，最近最少被使用)自动删除不使用的缓存
3. 对存储的数据设置过期时间，但是过期的数据采取惰性删除机制：不主动监控过期，而是访问的时候查看Key的时间戳，判断是否过期，过期返回空
4. 无集群模式
5. Memcached 采用 Slab Table 方式分配内存，可有效减少内存碎片，提升回收效率
6. 存储数据 Key 限制为250字节，Value 限制为 1MB，适用于小块数据的存储
7. 本身并不支持分布式，因此，一般在客户端通过一致性哈希这样的分布式算法来实现 Memcached 的分布式存储
8. 只支持键值对的存储，不支持其它数据结构，复杂的需要在应用程序自行处理
与 Redis 内存管理类似，Memcached 也没有直接采用 malloc/free 管理内存，而是采用 Slab Allocation 机制管理内存。
### Memcached高可用方案
Memcached 不支持真正意义上的集群模式，也不支持主从副本以防止单点故障。为了保障 Memcached 服务的高可用，需要借助第三方软件或者自己设计编程实现。
即 Memcached 在实现分布式群集部署时，Memcached 服务端之间是不能进行通讯的，也就是说服务端是伪分布式的，分布式将由客户端或者代理来实现。
#### 方案一：一致性Hash
当客户端向 Memcached 集群发送数据时，首先通过一致性哈希算法计算出该条数据的目标节点，然后将数据直接发送到该节点上存储。当客户端查询数据时，同样要计算出查询数据所在的节点，之后直接向该节点发送查询请求以获取数据。

通过一致性哈希算法可以保证数据存放到不同的 Mamcached 上，分散了在单台机器上的风险，提高了可用性，但只能解决数据全部丢失的问题，部分数据仍可能丢失，比如当一台 Mamcached 所在节点宕机，它上面的数据还是会丢失。
#### 方案二：Repcached
Repcached，全称 Replication Cached 高可用技术，简称复制缓冲区技术。Repcached 可用来实现 Memcached 的复制功能。它所构建的主从方案是一个单主单从方案，不支持多主多从。但是，主从两个节点可以互相读写，从而可以达到互相同步的效果。
假设主节点坏掉，从节点会很快侦测到连接断开，然后它会自动切换到监听状态（Listen）从而成为主节点，并等待新的从节点加入。
但原来挂掉的主节点恢复之后，只能作为从节点通过人工手动的方式重新启动。它并不能抢占成为新的主节点，除非新的主节点挂掉。这就意味着，基于 Repcached 实现的 Memcached 主从文案中，主节点并不具备抢占功能。
